Chapter 1 — Post-Training Quantization 2

1. What is Quantization (PTQ)?

Naively quantizing a trained model — without any additional training — is called Post-Training QuantizationPTQ — applying quantization after training is already complete. Fast and simple, but introduces rounding errors because the model's weights were never optimized to work well at the lower precision. The larger the model, the better PTQ tends to work; small models at 4-bit suffer more. (PTQ). The process for INT8 quantization, step by step:

1
Find the weight range
Compute max(abs(W)) — the largest absolute value in the weight matrix.
2
Compute the scale factor
Compute a = 127 / max(abs(W)). The value 127 is INT8's maximum range.
3
Quantize
Apply qW = int8(round(W × a)). Each weight is rounded to the nearest integer in [−127, 127].
4
Dequantize for inference
Reverse with float16(qW) / a. The rounding in step 3 introduces an error that cannot be recovered.

Why PTQ degrades accuracy

The rounding in step 3 is irreversible. At INT44-bit integer quantization — only 16 distinct values can represent a weight (-8 to 7). At this extreme compression, important fine-grained weight differences collapse to the same quantized value, causing the model to behave differently than the full-precision version. precision, weights can only take 16 distinct values. Important weight distinctions that the model learned during training are collapsed, and the model's behavior changes — sometimes significantly. This is the quantization gap that QAT is designed to close.

2. What is QAT & How It Works

Quantization-Aware TrainingQAT — a training technique where the quantization operation is simulated during the forward pass, making the model aware of and robust to the precision loss it will experience at inference. The model adjusts its weights during training to minimize the impact of quantization. (QAT) simulates the quantization procedure during training itself, rather than applying it after. The key idea is "fake quantization" — weights are rounded to the quantized representation and back during each forward pass, exposing the model to quantization error during gradient descent so it can learn to compensate.

Fake quantization explained

During each training step, the fake quantizeA differentiable approximation to quantization used during training. The weight is rounded to its quantized value and immediately dequantized back to float — so the computation stays in high precision (bfloat16), but the model "sees" the quantized version of its weights and learns to tolerate it. operation:

  1. Takes the full-precision weight W (in bfloat16)
  2. Rounds it to its quantized representation (e.g., the nearest INT4 value)
  3. Immediately dequantizes back to bfloat16
  4. Uses this "damaged" weight for the forward pass

The model receives gradients that account for the rounding damage and adjusts its weights to be more robust to it over thousands of training steps. At the end, the weights are converted to the actual quantized format for deployment.

The two-phase TorchAO pipeline

1
Insert fake quantize operations
TorchAOPyTorch Architecture Optimization — a PyTorch library for quantization and other model optimization techniques. Developed by Meta/PyTorch. Unsloth integrates TorchAO to provide QAT support, leveraging its fake quantize infrastructure and final conversion utilities. wraps each linear layer with fake quantize ops before training begins. The model trains normally, but every weight access goes through the fake quantize round-trip.
2
Convert to real quantized format
After training, TorchAO replaces the fake quantize ops with actual quantize/dequantize operations, producing a real INT4 or INT8 model ready for inference.

Chapter 3 — QAT + LoRA Fine-tuning 4

3. QAT + LoRA Fine-tuning

QAT in Unsloth combines with LoRALow-Rank Adaptation — adds small trainable adapter matrices to each weight layer. Only the adapter matrices are updated during training, keeping the base model weights frozen. Combined with QAT, the base model is fake-quantized while LoRA adapters train on top, giving both quantization robustness and task-specific adaptation. fine-tuning simultaneously. This means you get:

  • Quantization robustness from QAT — the model learns to tolerate its future INT4 form
  • Task-specific adaptation from LoRA — the model is simultaneously fine-tuned on your dataset
  • Memory efficiency from LoRA — only 1% of weights are trainable adapter parameters

Code — QAT + LoRA setup

from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/Qwen3-4B-Instruct-2507",
    max_seq_length = 2048,
    load_in_16bit = True,  # load full precision — QAT handles quantization
)

model = FastLanguageModel.get_peft_model(
    model,
    r = 16,
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj"],
    lora_alpha = 32,

    # Choose QAT scheme: "int4", "int8-int4", "fp8-int4", "fp8-fp8"
    qat_scheme = "int4",
)

# Train as normal with SFTTrainer — QAT operates transparently
Note on load_in_16bit. When using QAT, load the base model in 16-bit (load_in_16bit=True), not 4-bit. QAT needs the full-precision weights to fake-quantize during training. Using load_in_4bit with QAT would quantize before QAT can simulate it, defeating the purpose.

Chapter 4 — Exporting QAT Models 5

4. Exporting QAT Models

After fine-tuning, the model still contains fake quantize operations — they must be converted to real quantized format before inference. Unsloth provides model.save_pretrained_torchao for this. There are two steps.

Step 1 — Convert fake → real quantization

from torchao.quantization import quantize_
from torchao.quantization.qat import QATConfig

# This replaces fake quantize ops with real quantize/dequantize ops
quantize_(model, QATConfig(step = "convert"))

Step 2 — Save in your chosen format

Three options are shown below. The first is the most convenient — it reuses whatever QAT schemeThe quantization scheme used during training (e.g., int4, fp8-int4). Saved in model._torchao_config. Using base_config reuses exactly the same format that was trained — ensures consistency between training precision and inference precision. was used during training. The other two let you override the output format.

# Option A — match training scheme (recommended)
model.save_pretrained_torchao(
    model, "tokenizer",
    torchao_config = model._torchao_config.base_config,
)

# Option B — explicit INT4 output
from torchao.quantization import Int4WeightOnlyConfig
model.save_pretrained_torchao(
    model, "tokenizer",
    torchao_config = Int4WeightOnlyConfig(),
)

# Option C — explicit INT8 output
from torchao.quantization import Int8DynamicActivationInt8WeightConfig
model.save_pretrained_torchao(
    model, "tokenizer",
    torchao_config = Int8DynamicActivationInt8WeightConfig(),
)

After saving

The saved model is a standard TorchAOPyTorch Architecture Optimization format — the output format produced by save_pretrained_torchao. Compatible with vLLM, Unsloth inference, and other PyTorch-based inference engines. Can also be uploaded to the Hugging Face Hub. quantized model. It can be:

  • Served via vLLM for multi-user production inference
  • Loaded back in Unsloth for further fine-tuning or inference
  • Uploaded to the Hugging Face Hub with push_to_hub
  • Exported to ExecuTorch for mobile deployment (see page 6)
Uploading to Hugging Face. Remember to generate a Hugging Face token at huggingface.co/settings/tokens and add it to your environment before calling push_to_hub.

5. PTQ without Training & Mobile Deployment

Using save_pretrained_torchao for PTQ

You can call model.save_pretrained_torchao on any model without running QAT training — this performs standard PTQPost-Training Quantization — applying quantization after training without any QAT-aware fine-tuning. Faster than QAT but lower accuracy, especially at 4-bit. Useful when you want to convert an existing model to a TorchAO-compatible format without additional training. and saves the result in TorchAO format. Useful for converting existing models to FP8 or other formats for deployment on specialized hardware.

# Convert any loaded model to dynamic Float8 (no training required)
from torchao.quantization import PerRow
from torchao.quantization import Float8DynamicActivationFloat8WeightConfig

torchao_config = Float8DynamicActivationFloat8WeightConfig(
    granularity = PerRow()
)
model.save_pretrained_torchao(torchao_config = torchao_config)

FP8 for H100 / B200 inference

FP88-bit floating point — a newer number format supported natively by H100 and B200 Tensor Cores. Unlike INT8, FP8 retains a floating-point exponent, making it better suited for activations that vary widely in magnitude. FP8 inference on H100 can be significantly faster than FP16 with minimal accuracy loss. is a floating-point format supported natively on H100 and B200 GPUs. Unlike INT formats, FP8 retains an exponent, making it more accurate for activations. FP8 × FP8 matrix multiplications can be significantly faster than FP16 on compatible hardware.

ExecuTorch — mobile deployment

ExecuTorchPyTorch's framework for on-device inference — running LLMs directly on mobile phones, tablets, and edge devices without a server. Uses highly optimized kernels for ARM CPUs and mobile GPUs. Requires INT4 or INT8 quantization to fit models in the limited memory of mobile hardware. is PyTorch's solution for running LLMs on mobile devices (iOS, Android). QAT-trained models from Unsloth can be exported directly to ExecuTorch format, enabling on-device inference with the accuracy benefits of QAT.

A reference deployment is available at huggingface.co/metascroy/Qwen3-4B-int8-int4-unsloth — a Qwen3-4B model quantized via Unsloth QAT and ready for ExecuTorch deployment on mobile.
