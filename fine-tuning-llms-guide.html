Chapter 1 — What is Fine-tuning? 2

1. What Is Fine-tuning?

Fine-tuning — also called post-training — customizes a pre-trained model's behavior, enhances and injects new knowledge, and optimizes performance for specific domains or tasks.

For example, OpenAI's GPT-4 was post-trained to improve instruction following and helpful chat behavior. The standard method of post-training is called Supervised Fine-TuningSFT — the most common fine-tuning method. You provide labeled input-output pairs (e.g., question → answer) and the model learns to replicate them. Think of it as teaching by example. (SFT). Other methods include:

  • DPO / ORPODirect Preference Optimization / Odds Ratio Preference Optimization — preference-based fine-tuning methods where the model is trained on pairs of responses ranked by human preference, without needing a separate reward model. — preference optimization methods
  • Reinforcement LearningRL — an agent learns by interacting with an environment and receiving rewards or penalties. For LLMs, this means generating responses and scoring them with a reward function. GRPO and GSPO are popular RL variants for LLMs. (GRPO, GSPO) — reward-based learning
  • DistillationKnowledge distillation — training a smaller "student" model to mimic the outputs of a larger "teacher" model. Produces compact models that retain much of the teacher's capability. — compressing large model knowledge into smaller ones

By fine-tuning a pre-trained model on a dataset, you can:

  • Update + learn new knowledge — inject domain-specific information the base model never saw.
  • Customize behavior — adjust tone, personality, or response style.
  • Optimize for tasks — improve accuracy for specific use cases like legal analysis, customer support, or code generation.

Fine-tuning vs. RAG

A common misconception is that fine-tuning cannot teach new knowledge, or that RAGRetrieval-Augmented Generation — a technique that retrieves relevant documents at inference time and injects them into the prompt. RAG cannot change the model's weights; it only augments what the model sees. Fine-tuning changes the weights permanently. outperforms fine-tuning. This is false. Fine-tuning can replicate all of RAG's capabilities, but not vice versa — RAG can only augment what the model sees at inference time; it cannot change the model's weights.

What is LoRA / QLoRA?

Instead of updating all 70 billion numbers in a model like Llama 70B, LoRALow-Rank Adaptation — adds two thin matrices A and B alongside each weight matrix. Only A and B are trained, representing roughly 1% of total parameters. The original weights stay frozen. This makes fine-tuning fast and memory-efficient. adds two thin matrices A and B to each weight and optimizes only those — roughly 1% of all weights. The original model weights remain frozen.

QLoRAQuantized LoRA — combines LoRA adapters with 4-bit quantization of the base model. The base model is loaded in INT4 (saving 75% VRAM) while the LoRA adapters train in 16-bit. This enables fine-tuning very large models on consumer GPUs. extends LoRA by also quantizing the base model to 4-bit, saving 75% memory. This makes it possible to fine-tune large models on consumer hardware with as little as 3 GB VRAM.

Rule of thumb. Think of a fine-tuned model as a specialist — a generalist doctor who completed a residency in cardiology. The base model is the medical school education; fine-tuning is the residency.

Chapter 2 — Choose the Right Model & Method 3

2. Choose the Right Model & Method

If you are a beginner, start with a small instruct model like Llama 3.1 8B and experiment from there. You will also need to choose a training method.

Training method comparison

MethodWhen to useVRAM cost
QLoRAQuantized LoRA — trains LoRA adapters on top of a 4-bit quantized base model. Best starting point for most use cases. Unsloth's dynamic 4-bit quants recover most accuracy lost by standard INT4 quantization. Most use cases, recommended starting point Lowest (~3 GB for 7B)
LoRALow-Rank Adaptation — trains small adapter matrices on a 16-bit frozen base model. Slightly more VRAM than QLoRA but higher precision. When done correctly, LoRA can match full fine-tuning. When precision matters more than memory Medium (16-bit base)
SFTSupervised Fine-Tuning — the standard method. You provide labeled examples (question → answer pairs) and the model learns from them. Works for most tasks: chatbots, classification, extraction, summarization. (Full fine-tune) Rarely needed; try LoRA first Very high
RL / GRPOReinforcement Learning / Group Relative Policy Optimization — used when you need the model to excel at a specific behavior (e.g., tool-calling, math reasoning) by optimizing a reward function rather than labeled examples. Specific behavior mastery (reasoning, tool-calling) Medium–High
Research insight. Training and serving in the same precision helps preserve accuracy. If you plan to serve in 4-bit, train in 4-bit — and vice versa.

Instruct vs. base models

Instruct modelsModels already fine-tuned for instruction following and conversation. They understand chat templates (ChatML, ShareGPT) and require less data to further fine-tune. Recommended for beginners. allow direct fine-tuning using conversational chat templates and require less data. Base modelsRaw pre-trained models with no instruction tuning. They require more data and careful prompt formatting (Alpaca, Vicuna templates) to fine-tune effectively. Better for pretraining-style tasks or building a new chat model from scratch. require more data and careful formatting — use them only when building from scratch.

Key model settings

ParameterDefaultWhat it controls
max_seq_length2048Context window during training. Start low; increase if needed.
load_in_4bitTrueEnables QLoRA (4-bit base model). Set False for LoRA 16-bit.
dtypeNoneAuto-detects. Set bfloat16 explicitly for newer GPUs.
full_finetuningFalseEnables full fine-tune (FFT). Almost never needed.
Common mistake. Jumping straight to full fine-tuning (FFT) because LoRA seems insufficient. Start with QLoRA. If it fails, LoRA won't save it either — and neither will FFT. The problem is almost always the dataset.

Chapter 3 — Your Dataset 4

3. Your Dataset

For LLMs, datasets are collections of text data used to train the model. The quality and quantity of your dataset will largely determine the outcome of your fine-tune — this is the most important step.

Basic structure

Most fine-tuning datasets require at minimum two columns: a question (or instruction) and an answer (or response). This is called a QA pairQuestion-Answer pair — the fundamental unit of supervised fine-tuning data. The model learns to map a given input (question/instruction) to a desired output (answer/response). Quality of QA pairs matters far more than sheer quantity..

{ "instruction": "Summarize this legal contract section.",
  "input": "Party A agrees to pay Party B...",
  "output": "Party A must pay Party B within 30 days..." }

How to create your dataset

  • Synthetically generate data using ChatGPT or a local LLM to create structured QA pairs from raw documents.
  • Use Unsloth's Synthetic Dataset notebook — automatically parses PDFs, videos, and documents, generates QA pairs, and auto-cleans using Llama 3.2.
  • Curate manually for highest quality. Even 200–500 high-quality examples can outperform 10,000 noisy ones.

Format by use case

Use caseRecommended formatNotes
Chatbot / assistantChatML / ShareGPTMulti-turn conversation format
Instruction followingAlpacainstruction + input + output triplets
Code generationRaw code dumpsStructure less critical for code tasks
ClassificationQA pairsLabel set in answer column
Vision / multimodalImage + text pairsImages embedded in answer output
Exception for code. Dumping raw code without QA structure can still yield significant improvements for code-focused fine-tunes. The structured QA recommendation applies primarily to conversational and instruction-following tasks.

Dataset size guidance

There is no universal rule, but as a rough starting point: for a LoRALow-Rank Adaptation fine-tune — because only a small fraction of weights are updated, LoRA requires far less data than full fine-tuning to achieve good results. 500–5,000 high-quality examples is a common effective range. fine-tune, 500 to 5,000 high-quality QA pairs is a reasonable starting range. Quality consistently outperforms quantity.

Chapter 4 — Training Hyperparameters 5

4. Training Hyperparameters

HyperparametersSettings that control how training proceeds — not the model weights themselves. Examples: learning rate, batch size, number of epochs. They are set before training and tuned through experimentation. are settings you configure before training begins. They control speed, memory usage, and how well the model learns from your data.

LoRA-specific parameters

ParameterTypical valueEffect
r (rank)8 – 64 LoRA rankControls the size of the A and B adapter matrices. Higher rank = more parameters trained = more expressive but more memory. Start at r=16; increase only if the model underfits. — higher = more capacity, more memory.
lora_alpha= r or 2r Scaling factor for LoRA updates. Set equal to rank as a starting point.
lora_dropout0 – 0.1 DropoutRandomly zeroes out a fraction of activations during training to prevent overfitting. A value of 0.05 means 5% of activations are dropped per step. Set to 0 for small datasets to preserve signal. rate on LoRA layers. Use 0 for small datasets.
target_modulesq, k, v, o, gate Which weight matrices to apply LoRA to. More modules = more capacity.

Training loop parameters

ParameterDefaultEffect
per_device_train_batch_size2 Batch sizeNumber of training examples processed per gradient update step. Larger batch = more stable gradients but more VRAM. Prefer increasing gradient_accumulation_steps over batch size to avoid padding overhead. per GPU. Increase carefully to avoid padding waste.
gradient_accumulation_steps4 Gradient accumulationSimulates a larger batch size by accumulating gradients over N steps before updating weights. Effective batch = batch_size × accumulation_steps. Keeps VRAM low while stabilizing training. — simulates larger batch without extra VRAM.
learning_rate2e-4 Learning rateControls how large each weight update step is. Too high = unstable training (loss spikes). Too low = very slow convergence. For LoRA fine-tuning, 1e-4 to 2e-4 is a common safe range. Reduce to 5e-5 or 2e-5 for more precise fine-tuning. — lower = slower but more precise. Try 1e-4, 5e-5, 2e-5.
num_train_epochs1 – 3 EpochsOne full pass through the entire training dataset. 1–3 epochs is typical for fine-tuning to avoid overfitting. More epochs on small datasets risks memorization rather than generalization. — 1–3 recommended to avoid overfitting.
max_steps60 Hard cap on training steps. Use num_train_epochs for full runs.
warmup_steps5 – 10% of steps Gradually ramps up learning rate at start to stabilize early training.
Start conservative. Use the defaults for your first run. Only tune hyperparameters once you have a baseline result — change one variable at a time so you can tell what actually helped.

Chapter 6 — Training & Evaluation 7

6. Training & Evaluation

Once setup is complete, run the training cell. You will see a live log of the training lossA number measuring how wrong the model's predictions are on the training data. It should decrease over time. A final loss of 0.5–1.0 is typical for conversational fine-tuning. Loss of 0 means the model memorized the training data (overfitting). — a number that should decrease as training progresses.

Reading the training loss

Loss rangeInterpretationAction
> 2.0Model not learningCheck dataset format; increase learning rate
1.0 – 2.0Early learning stageNormal for first steps; wait for convergence
0.5 – 1.0Good learning progressTypical healthy final loss for most tasks
0.1 – 0.5Good — watch validationEnable validation set to check for overfitting
≈ 0.0OverfittingReduce epochs; increase dropout; add more data

Evaluation approaches

Manual evaluation — chat with the model and judge responses. Simple but effective for conversational tasks. Always start here.

Held-out validation set — reserve 20% of your training data before training and evaluate on it during or after training. Configure with evaluation_steps = 100 to avoid slowdowns.

Automated evaluation — tools like EleutherAI's LM Evaluation HarnessAn open-source framework for benchmarking LLMs on standardized tasks (e.g., MMLU, HellaSwag, HumanEval). Useful for comparing model versions, but automated benchmarks may not align with your specific task requirements. run standardized benchmarks. Use as a supplement to manual evaluation, not a replacement.

Common training problems

SymptomLikely causeFix

Loss spikes / NaNLearning rate too highLower learning rate; add warmup steps
VRAM OOM errorBatch size too largeReduce batch size; use gradient accumulation
Model ignores instructionsWrong chat templateMatch template to model (ChatML vs Alpaca)
Loss → 0 quicklyDataset too small / repeatedAdd data; reduce epochs

7. Running & Deploying the Model

After training, always call FastLanguageModel.for_inference(model) before running inference. Unsloth provides 2× faster inference natively — this call activates it.

Saving the LoRA adapter

The fine-tuned model is saved as a small LoRA adapterA small file (typically 50–200 MB) containing only the trained A and B adapter matrices. It is used on top of the original base model at inference time. Much smaller than saving the full model (which would be 14+ GB for a 7B FP16 model). — typically 50–200 MB — rather than the full model weights.

# Save LoRA adapter locally
model.save_pretrained("my-lora-adapter")
tokenizer.save_pretrained("my-lora-adapter")

# Or push to Hugging Face Hub
model.push_to_hub("your-username/my-model", token="hf_...")

Inference settings

Adjust max_new_tokens to control response length. Default is 128 — increase to 256 or 1024 for longer outputs. A higher value will proportionally increase generation time (each additional token requires another full decode step at the TBTTime Between Tokens — the interval between each output token, equal to 1 ÷ TPS. Multiplying by max_new_tokens gives the approximate total generation time. rate).

Chapter 8 — Summary & Next Steps 9

8. Summary & Next Steps

You have covered the complete fine-tuning workflow: understanding what fine-tuning does, choosing a method, preparing a dataset, setting hyperparameters, training, evaluating, and deploying.

The fine-tuning checklist

  1. Choose an instruct modelA model already instruction-tuned — it understands conversational prompts and chat templates. Easier to fine-tune further than a raw base model because it already knows how to follow instructions. appropriate for your task and hardware.
  2. Start with QLoRAQuantized LoRA — the recommended starting method. Trains adapters on a 4-bit base model. Requires the least VRAM and is sufficient for most fine-tuning goals. before considering LoRA full-precision or FFT.
  3. Prepare a clean, well-structured dataset of QA pairs. Quality over quantity.
  4. Use default hyperparameters for the first run. Tune only after establishing a baseline.
  5. Monitor training lossShould decrease steadily from ~2.0 to 0.5–1.0 over the course of training. A stalled loss means the model is not learning. A loss near 0 means overfitting. — it should decrease and stabilize, not reach zero.
  6. Evaluate manually first; add a validation set for quantitative measurement.
  7. Save as a LoRA adapter and deploy to your target inference engine.

QLoRA (INT4 base)LowestGoodFast
LoRA (FP16 base)MediumBetterMedium
Full fine-tuneHighestBestSlow
