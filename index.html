Part II — Token Generation Fundamentals 3

II. How Token Generation Works

LLM inference is autoregressive: each tokenSmallest text unit processed by an LLM — roughly 0.75 words. "ChatGPT" = 1 token; "unbelievable" ≈ 3 tokens. is generated one at a time by performing a full forward pass through the model. This differs fundamentally from training, where many samples are processed together.

Two phases of inference

Prefill. All input prompt tokens are processed in parallel. This phase is compute-bound — the hardware is busy doing matrix multiplies. It ends by building the KV cacheKey-Value cache: stores the computed attention states for every prior token so they do not need to be recomputed at each decode step. Grows linearly with context length., which stores the "memory" of the entire prompt.

Decode. One token is generated per step. Each step loads the full model weights from VRAMVideo RAM — high-speed memory physically on the GPU. Holds model weights, KV cache, and activations. H100 has 80 GB. to produce a single output. This phase is memory-bandwidthMemory bandwidth (GB/s): how fast the GPU can read data from VRAM to its compute cores. H100 = 3,350 GB/s. This — not TFLOPS — is usually the bottleneck during decode.-bound for typical batch sizes.

The Roofline model

Every hardware setup has two ceilings. The roofline modelA framework that shows which resource — compute (TFLOPS) or memory bandwidth (GB/s) — limits performance first. The actual bottleneck depends on the arithmetic intensity of the workload. tells you which one applies:

Compute ceiling: Max TPS = Tensor_TFLOPS × 10¹² ÷ FLOPs_per_token
Bandwidth ceiling: Max TPS = Mem_BW_GBps ÷ (Model_GB ÷ Batch_size)

Actual TPS ≈ min(Compute_ceiling, Bandwidth_ceiling) × MFU

For decode at batch sizeNumber of requests processed simultaneously. Batch 1 = one user; batch 32 = 32 users in parallel. Higher batch amortizes weight-loading cost, improving throughput. ≤ 32, virtually all modern GPUs are bandwidth-bound, not compute-bound.

FLOPs per token

For a decoder-only transformer in the decode phase:

FLOPs_per_token ≈ 2 × N_params

The factor 2 comes from one multiply and one add per parameter (a fused multiply-add, or FMA). For a 7 B model this is 14 × 10⁹ FLOPs per token.

The ridge point

The ridge pointThe batch size at which arithmetic intensity crosses from memory-bound into compute-bound territory. Below it, more bandwidth helps. Above it, more TFLOPS help. Ridge point = TFLOPS × 1000 ÷ BW_GBps. is the batch size at which your workload crosses from bandwidth-limited to compute-limited:

Ridge_point = Tensor_TFLOPS × 1000 ÷ Mem_BW_GBps

H100 SXM example: 989 × 1000 ÷ 3350 ≈ 295
→ Compute-bound only at batch ≥ 295 (FP16)
Key insight. For single-user or small-batch serving, buying a GPU with more TFLOPSTera FLOPs per second — 10¹² arithmetic operations per second. The peak compute rating of a GPU. For LLMs, always use Tensor Core TFLOPS, not CUDA Core TFLOPS. rarely helps. Buying one with more bandwidth does.

Part III — CPU-Based Token Estimation 4

III. CPU-Based Token Estimation

CPU inference is used for edge deployments, quantized small models (≤ 13 B), or when a GPU is unavailable. Performance is almost always limited by DDR memory bandwidth, not compute.

Core formula

TPS_cpu = Mem_BW_GBps ÷ (Model_GB ÷ Batch_size) × Efficiency

Where:
Model_GB = N_params × bytes_per_weight
Efficiency = 0.70 – 0.85 (accounting for memory latency & contention)

Bytes per weight by precisionThe numeric format used to store model weights. FP32 = 4 bytes (full precision); FP16/BF16 = 2 bytes (half); INT8 = 1 byte; INT4 = 0.5 bytes. Lower precision = smaller model = faster on bandwidth-bound hardware.:

FormatBytes per weightNotes
FP324Full precision, rarely used for inference
FP16 / BF162Standard inference precision
INT81Halves VRAM; minor quality loss
INT40.5Quarters VRAM; acceptable for most uses

Multi-core scaling

Additional cores help only when the workload is not already saturating the memory bus. HyperthreadingIntel/AMD feature that runs two hardware threads per physical core. For memory-bound LLM inference it adds roughly 10–30% throughput — not 2×, because both threads share the same memory bus. adds roughly 10–30% for memory-bound inference — not the 2× that the thread count suggests.

Worked example

Setup: Llama 3 8 B quantized to INT44-bit integer quantization. Each weight stored in 0.5 bytes instead of 2 bytes (FP16). Reduces memory by 4×, enabling large models on CPU or consumer GPUs. Minor quality degradation measured in perplexity., AMD EPYC 9654, DDR5 @ 460 GB/s.

Model size in RAM = 8 × 10⁹ × 0.5 bytes = 4.0 GB
Bytes loaded per token = 4.0 GB (full weight sweep each step)
Max TPS (BW ceiling) = 460 ÷ 4.0 = 115 tok/s
Apply 0.75 efficiency → ≈ 86 tok/s at batch 1

Practical notes

  • Use llama.cpp with AVX-512 or ARM AMX for best CPU performance.
  • NUMA-aware memory allocation matters on dual-socket servers — keep model and threads on the same socket.
  • CPU TDPThermal Design Power — the maximum sustained wattage a chip draws under full load. Server CPUs range from 120 W to 500 W. Multiply by number of sockets for total CPU power. for server CPUs ranges from 150 W (efficiency) to 500 W (high-core-count). Plan cooling accordingly.

Part IV — GPU-Based Token Estimation 5

IV. GPU-Based Token Estimation

CUDA cores vs Tensor cores

CUDA coresGeneral-purpose compute units on an NVIDIA GPU. Handle FP32 and INT32 arithmetic — everything that is not a matrix multiply. Do NOT use CUDA Core TFLOPS to estimate LLM speed. handle general arithmetic. Tensor coresSpecialized GPU units for matrix multiplication (GEMM). Up to 16× faster than CUDA cores for the same operation. Since 95%+ of transformer compute is matmul, Tensor Core TFLOPS is the correct spec to use for LLM benchmarking. are dedicated matrix-multiply units — up to 16× faster for that operation. Since transformers are 95%+ matrix multiplications, always use Tensor Core TFLOPS, not CUDA Core TFLOPS, when estimating LLM performance.

GPU throughput formula

TPS_gpu = min(Compute_TPS, Bandwidth_TPS) × MFU

Compute_TPS = (Tensor_TFLOPS × 10¹²) ÷ (2 × N_params)
Bandwidth_TPS = Mem_BW_GBps ÷ (N_params × dtype_bytes ÷ Batch_size)

MFU (typical):
Naive PyTorch → 0.10 – 0.25
vLLM / TensorRT-LLM → 0.35 – 0.55

MFUModel FLOPs Utilization — the fraction of a GPU's theoretical peak that is actually used. A GPU rated at 1,000 TFLOPS running at 40% MFU delivers 400 effective TFLOPS. Always apply MFU before trusting a throughput estimate. accounts for all real-world losses: kernel launch overhead, memory latency, tensor padding, and operator fusion gaps. Never assume MFU = 1.0.

Effect of batch size

Increasing batch sizeNumber of concurrent requests processed together. Higher batch amortizes the cost of loading weights from VRAM over more users, improving throughput at the cost of higher per-request latency. raises arithmetic intensity until the ridge point is crossed:

Arithmetic_intensity = 2 × Batch_size ÷ dtype_bytes
GPU is compute-bound when: Arithmetic_intensity > Ridge_point

→ Below ridge: add bandwidth (faster GPU or fewer parameters)
→ Above ridge: add compute (more TFLOPS or larger batch)

Multi-GPU tensor parallelism

Tensor parallelismSplitting one model layer across N GPUs. Each GPU holds 1/N of the weight matrices and computes in parallel. Requires high-speed GPU interconnects (NVLink preferred) to exchange partial results between steps. distributes one model across N GPUs:

TPS_N_GPU ≈ TPS_1_GPU × N × Parallel_efficiency

Parallel_efficiency:
NVLink (H100 SXM) → 0.70 – 0.90
PCIe → 0.50 – 0.70

NVLinkNVIDIA's high-speed GPU-to-GPU interconnect (~900 GB/s). Tensor parallelism requires GPUs to exchange activation tensors between every transformer layer. NVLink makes this fast; PCIe makes it a bottleneck at TP ≥ 4. efficiency drops as N grows because inter-GPU communication increases with model depth. Beyond TP = 8, returns diminish steeply.

Part V — VRAM Requirements 6

V. VRAM Requirements

Total VRAM formula

VRAM_total = VRAM_weights + VRAM_KV_cache + VRAM_overhead

VRAM_weights = N_params × dtype_bytes
VRAM_KV_cache = 2 × N_layers × D_model × dtype_bytes × ctx_len × batch
VRAM_overhead ≈ 15 – 20 % of weights (CUDA context, fragmentation)

Rule of thumb: multiply your minimum by 1.20 to account for overhead.

The factor of 2 in the KV cacheKey-Value cache — stores both the Key and Value matrices for every attention layer for every past token. The 2× factor reflects that both K and V are stored. Grows linearly with context length and batch size. formula represents both the Key and Value matrices.

KV cache growth with context

For Llama 3 70 B at FP16 (N_layers = 80, D_model = 8192):

KV per token = 2 × 80 × 8192 × 2 bytes = 2,621,440 bytes ≈ 2.5 MB

At context 4,096 tokens, batch 1: 4,096 × 2.5 MB ≈ 10.2 GB
At context 8,192 tokens, batch 4: 32,768 × 2.5 MB ≈ 82.0 GB

At batch 4 and 8 K context, the VRAMVideo RAM — the fast on-chip memory of the GPU. Everything the model needs during inference must fit here. If it does not fit, you must quantize, reduce batch size, shorten context, or add GPUs. needed for the KV cache alone (82 GB) exceeds the model weights (140 GB ÷ 2 for INT8 = 70 GB).

Part VI — Electricity Consumption 7

VI. Electricity Consumption

Instantaneous power draw

P_total = (P_GPU × N_GPU + P_CPU + P_DRAM + P_NVMe) × Utilization

P_GPU = GPU TDP in watts
Utilization = 0.70 – 1.00 for active inference
P_CPU = 150 – 500 W (server CPUs)
P_DRAM ≈ 3 W per 32 GB DIMM
P_NVMe ≈ 5 – 10 W per drive

TDPThermal Design Power — the maximum sustained wattage a chip draws under full load. Real-world power under typical inference load is 70–95% of TDP. Use TDP for worst-case planning; use 80% for average cost estimates. is the worst-case figure. Real inference workloads typically draw 70 – 95% of TDP.

Energy per token

Energy per token (Joules) = P_total_watts ÷ TPS
Energy per token (Wh) = P_total_watts ÷ (TPS × 3,600)

Example — H100 SXM serving Llama 3 70 B:

P_total = 700 W TDP × 0.90 utilization = 630 W
TPS = 2,000 tok/s (batch 16, TP=4, vLLM)
Energy = 630 ÷ 2,000 = 0.315 J / token

Cost per million tokens

Cost / M_tokens = (J_per_token × 1,000,000 ÷ 3,600,000) × $/kWh × PUE

Example (above, $0.10/kWh, PUE 1.3):
= (0.315 × 10⁶ ÷ 3.6 × 10⁶) × 0.10 × 1.3
= 0.0875 kWh × 0.10 × 1.3
= $0.0114 per 1 M tokens (power only)

PUEPower Usage Effectiveness — a data-center overhead multiplier. PUE 1.0 would mean zero overhead (physically impossible). Modern hyperscale DCs achieve 1.1 – 1.2; typical colocation is 1.3 – 1.5; older facilities can be 1.8+. Always include PUE in electricity cost estimates. accounts for cooling, UPS, lighting, and networking overhead — never assume the wall-meter reads only server wattage.

Monthly server power cost

Monthly_kWh = P_total_kW × 720 h × Avg_utilization × PUE
Monthly_cost = Monthly_kWh × $/kWh

Example (4× H100, 2.8 kW, 70% util, PUE 1.3, $0.10/kWh):
= 2.8 × 720 × 0.70 × 1.3 × $0.10 = $183 / month
Remember. Power cost is only part of total cost. Add hardware amortization (CapExCapital Expenditure — the upfront cost of purchasing hardware. Typically amortized over 3 years for GPU servers. Example: a $40,000 H100 server = $1,111/month CapEx over 36 months.) and colocation fees to arrive at true cost-per-token.

Part VII — Hardware Backtracking 8

VII. Hardware Selection by Backtracking

Backtracking means starting from your SLA requirements and working backwards to the minimum hardware that satisfies them.

Inputs (your requirements)

RequirementExample
Target TPSTokens per second — your throughput SLA. This is the primary number that drives hardware selection. Setting it too low wastes money; too high wastes capacity.500 tok/s sustained
Max TTFTTime to First Token — the latency from sending a prompt to seeing the first output word. Drives user experience for interactive applications. Long contexts increase TTFT significantly.200 ms
Context length8,192 tokens
Model size70 B parameters
PrecisionFP16 or INT8
Power budget3.5 kW

Step-by-step procedure

Step 1 — Compute VRAM floor. Calculate the minimum VRAMVideo RAM — must hold weights + KV cache + overhead. If total exceeds single-GPU VRAM, you need multiple GPUs or quantization. needed for weights plus KV cache at your target context and batch size, multiplied by 1.20 for overhead.

Step 2 — Derive required bandwidth. Required_BW = model_GB × TPS ÷ batch_size. This is usually the binding constraint. Select GPUs whose combined memory bandwidthGB/s of data transfer from VRAM to compute units. For bandwidth-bound inference (most small-batch cases), throughput scales almost linearly with bandwidth. meets or exceeds this figure.

Step 3 — Verify compute headroom. Confirm that N_GPU × Tensor_TFLOPS × MFU ≥ (2 × N_params × TPS) / 10¹². If not, increase GPU count or use a higher-TFLOPS card.

Step 4 — Check power budget. N_GPU × GPU_TDP ≤ Power_budget / 1.3. If over budget, apply quantizationReducing weight precision to cut VRAM and bandwidth requirements. INT8 halves bandwidth need; INT4 quarters it — enabling the same TPS target on fewer or lower-TDP GPUs.: INT8 halves bandwidth need, INT4 quarters it.

Step 5 — Check TTFT. TTFT = Prompt_length × FLOPs_per_token ÷ (TFLOPS × MFU × 10¹²). If this exceeds your budget, either increase GPU compute or impose a maximum prompt length.

Step 6 — Validate total cost. Compute monthly OpEx (power) + amortized CapEx, then divide by expected tokens/month to get $/token.

When Step 4 fails, do not automatically add GPUs. Try quantization first — INT4 often allows the same TPS on half the GPU count at one-quarter the bandwidth cost, yielding a better price-to-performance ratio.

Part IX — Worked Examples 10

IX. Worked Examples

Example A — Consumer GPU chatbot (7 B, single GPU)

Requirements: 50 tok/sTokens per second — the throughput target. 50 tok/s is comfortable for a single interactive user seeing text stream in real-time., 4,096 context, Llama 3 8 B, < $500/month.

Step 1 — VRAM: 8B × 0.5 bytes (INT4) = 4 GB weights. KV at batch 4, 4K ctx: 4 × 4096 × 0.5 MB = 8.2 GB. Overhead 0.8 GB. Total ≈ 13 GB → RTX 4060 Ti 16 GB fits.

Step 2 — BW check: Required = 4 GB × 50 ÷ 4 = 50 GB/s. RTX 4060 Ti: 288 GB/s >> 50. ✓

Step 3 — Power: 165 W × 720 h × 1.1 × $0.12 = $15.6/month. ✓

Result: RTX 4060 Ti 16 GB with INT4 via llama.cpp → ~100 tok/s actual (2× headroom).

Example B — Production API (70 B, data center)

Requirements: 2,000 tok/s, 8,192 context, Llama 3 70 B FP16, < 4 kW.

Step 1 — VRAM: 140 GB weights + ~82 GB KV (batch 4, 8K, INT8 KV) + 30 GB overhead = 252 GB → 4× H100 SXM (320 GB). Use paged attention.

Step 2 — BW check: Required per GPU = 140 GB × 2,000 ÷ (16 × 4) = 10,937 GB/s per GPU. H100 at 3,350 GB/s — needs batch ≥ 4 to amortize. At batch 16: feasible. ✓

Step 3 — Power: 4 × 700 W = 2.8 kW + 0.6 kW server = 3.4 kW < 4 kW. ✓

Result: 4× H100 SXM, TensorRT-LLM, TP=4, INT8 KV cache → ~2,200 tok/s.

Example C — CPU-only edge server (7 B)

Requirements: 20 tok/s offline, Mistral 7 B INT4, no GPU.

Model size = 7B × 0.5 bytes = 3.5 GB
Required BW = 3.5 GB × 20 = 70 GB/s
AMD EPYC 9654: 461 GB/s >> 70 GB/s
Expected TPS = 461 ÷ 3.5 × 0.80 ≈ 105 tok/s (5× headroom)

Result: Any modern server CPU suffices. Use llama.cpp with AVX-512.
