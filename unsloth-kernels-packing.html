Chapter 1 — Fused QK RoPE Triton Kernel 2

1. Fused QK RoPE Triton Kernel

Rotary Position Embedding (RoPERotary Position Embedding — a method of encoding token positions into the attention mechanism by rotating the Query and Key vectors by an angle proportional to their position in the sequence. Used in Llama, Qwen, Mistral, and most modern LLMs.) is applied to the Query (Q) and Key (K) vectors in every attention layer before the attention dot product. It is a hot path in training — called once per layer per batch.

What changed

Previously, RoPE required two separate Triton kernelsGPU kernel — a function that runs in parallel across thousands of GPU threads simultaneously. Each kernel launch has overhead. Merging two kernels into one eliminates half the launch overhead and allows the compiler to optimize the fused operation more aggressively. — one for Q and one for K. Unsloth merged them into a single fused kernel and added variable-length RoPE support, which is required for padding-free and packing.

  • 2.3× faster on long-context lengths in micro-benchmarks
  • 1.9× faster on shorter context lengths
  • Fully in-place — all tensor clones and contiguous transpose operations eliminated, reducing GPU memory allocation
  • Variable-length support enables correct position ID reset when multiple packed sequences are concatenated

Mathematical basis

The forward pass applies RoPE as a rotation. The backward pass (gradient computation) uses the same rotation matrix transposed. Unsloth exploits that the transpose of the rotation matrix R has the same structure — only the sign of the sine component flips — allowing the backward pass to reuse the same kernel with sin = -sin:

Forward:
  Q * cos + rotate_half(Q) * sin
  ≡ Q * cos + Q @ R * sin
  where R = [ 0,  I]
             [-I,  0]

Backward (gradient dC/dY):
  dY * cos + dY @ R.T * sin
  where R.T = [ 0, -I]   ← same structure, sign flipped
               [ I,  0]

∴ backward = forward with  sin1 = -sin1  (in-place flip)

This means the backward kernel is not a separate implementation — it reuses the forward kernel with a sign flag, cutting code surface area and ensuring both paths benefit from the same optimizations.

History. The original RoPE kernel was introduced in December 2023 at Unsloth's launch. In March 2024, a community contribution (PR 238) added head-grouped block launching, gaining 1–2% end-to-end speedup. The current fused QK version is the third generation — the most significant improvement yet.

Chapter 2 — Int64 Indexing for Long Context 3

2. Int64 Indexing for Long Context

During 500K long-context training runs, a class of CUDA out-of-bounds errors appeared in the MLP kernels (SwiGLUSwish-Gated Linear Unit — the activation function used in the feed-forward (MLP) layers of Llama, Qwen, and many other modern transformers. SwiGLU(x, g) = x * sigmoid(g) * g. Unsloth has a custom Triton kernel for it. and GeGLUGELU-Gated Linear Unit — a variant of gated activation used in some transformer MLP layers. Like SwiGLU but with GELU instead of Swish. Also has a custom Triton kernel in Unsloth.). The root cause was 32-bit integer indexing.

The problem

By default, TritonA GPU programming language/compiler developed by OpenAI. Triton defaults to 32-bit integer indexing for GPU memory offsets. At 32-bit, the maximum addressable index is 2³² ≈ 4.3 billion — which can overflow at very long sequence lengths with large hidden dimensions. and CUDA use 32-bit integer (int32) indexing for memory offsets. The maximum value representable in int32 is approximately 4.3 billion. At sequence lengths of 500K tokens with large hidden dimensions, the flattened tensor index can exceed this limit, causing silent memory corruption or crashes.

The solution

A naïve fix — converting all indices to int64 — would slow down shorter training runs because int64 arithmetic is heavier on GPU hardware. Unsloth instead uses a compile-time constanttl.constexpr in Triton — a value known at kernel compilation time. The Triton compiler generates specialized code for each value, so LONG_INDEXING=True compiles a different (int64) version than LONG_INDEXING=False (int32). No runtime branching overhead. flag LONG_INDEXING: tl.constexpr so the compiler generates two specialized versions of the kernel — one with int32 (fast, for normal runs) and one with int64 (correct, for long-context runs):

block_idx = tl.program_id(0)

if LONG_INDEXING:
    offsets = block_idx.to(tl.int64) * BLOCK_SIZE \
              + tl.arange(0, BLOCK_SIZE).to(tl.int64)
    n_elements = tl.cast(n_elements, tl.int64)
else:
    offsets = block_idx * BLOCK_SIZE \
              + tl.arange(0, BLOCK_SIZE)

The Triton compiler specializes on the constant value — there is no runtime branch and no performance cost for short-context runs.

When is LONG_INDEXING activated?

ConditionIndexing modeEffect
Context ≤ ~128K tokens (typical)int32Maximum performance, no overhead
Context > ~128K tokens (long-context fine-tune)int64Correct results, marginal overhead
Practical note. For most fine-tuning jobs — chatbots, instruction following, coding assistants — the standard int32 path is used automatically. Long-context int64 mode activates only when the product of sequence length × hidden dimension × batch size exceeds the int32 ceiling.

Chapter 3 — Why Padding is Wasteful 4

3. Why Padding is Wasteful — the Math

GPUs process all items in a batchA group of training examples processed together in one GPU forward/backward pass. All sequences in a batch must be the same length — so short sequences are padded with zeros to match the longest one. Those zero-padded positions compute nothing useful but still consume GPU cycles. at the same sequence length simultaneously. When a batch contains sequences of different lengths, every sequence is padded with zeros to match the longest one. Those padding tokens consume compute and memory but contribute nothing to learning.

Worst-case analysis

Assume a dataset of 50% short sequences (length S) and 50% long sequences (length L). Without packing, the batch must accommodate the longest sequence, so total token usage per batch is:

Token usage (no packing) = batch_size × L

With packing, short and long sequences are concatenated into one long 1D tensor per batch row. Token usage becomes:

Token usage (packed) = (batch_size / 2) × L + (batch_size / 2) × S

Speedup formula

Speedup = (batch_size × L) / ((batch_size/2 × L) + (batch_size/2 × S)) = 2L / (L + S) If S → 0 (very short sequences): Speedup → 2L / L = 2× If dataset is 80% short (20% long, 80% short): Speedup = L / (0.2L + 0.8S) → L / 0.2L = 5×
Key insight. The speedup from packing is not fixed — it depends on how short the short sequences in your dataset are. A dataset of very short QA pairs (e.g., single-sentence answers) benefits far more than a dataset of uniformly long documents.

Padding waste grows with batch size

A counterintuitive fact: increasing batch sizeWithout packing, increasing batch size increases the chance that a very long sequence appears in the batch — causing all other sequences to pad to that length. At batch size 8 on typical datasets, nearly 50% of all processed tokens are padding. Packing eliminates this waste entirely. without packing makes this worse, not better. At batch size 8 on typical conversational datasets, roughly 50% of tokens processed are padding zeros. Packing maintains near-100% valid token density at any batch size.

4. Padding-Free by Default

Since the latest Unsloth update, padding-free batchingA batching strategy where sequences are trimmed to the actual maximum length in each batch, rather than padded to a fixed global maximum. This eliminates padding tokens from every batch without requiring full sequence packing. Enabled automatically in the latest Unsloth — no code changes needed. is automatically enabled for all training runs. No configuration is required. This alone delivers:

2×
Training speed for Qwen3-8B and Qwen3-32B
60%
Memory usage reduction for Qwen3 models
0
Change in loss curve or gradient norms

What padding-free means

In standard batching, all sequences are padded to the same length — either a fixed max_seq_length or the longest sequence in the batch. Padding-free batching trims each batch to the actual maximum sequence length present, eliminating all trailing zeros. The attention mask is still respected correctly — this is not removing attention masking.

Chapter 5 — Uncontaminated Packing 6

5. Uncontaminated Packing

Standard sequence packing — concatenating multiple short sequences into one long tensor — causes a correctness problem: the attention mechanismThe core operation in a transformer. Each token attends to (looks at) every other token in its sequence by computing dot products between Query and Key vectors. In a packed sequence, without special masking, a token from sequence 2 would incorrectly attend to tokens from sequence 1. will allow tokens from one example to attend to tokens from a different packed example. This is called attention contamination, and it changes what the model learns.

How Unsloth prevents contamination

Unsloth's packing implementation preserves sequence length metadataWhen multiple sequences are packed into one 1D tensor, their individual start and end positions are stored separately. The attention kernel uses this metadata to build a block-diagonal attention mask — each sequence can only attend to its own tokens. alongside the packed tensor. This metadata is used to construct a block-diagonal attention mask so each sequence can only attend to its own tokens. Additionally, position IDs are reset to zero at the start of each packed sequence using the variable-length RoPE kernel described in Chapter 1.

The two conditions for correct packing

  1. Attention masking — block-diagonal mask prevents cross-sequence attention. Each example sees only its own context.
  2. Position ID reset — RoPE position IDs reset to 0 at the start of each packed sub-sequence. Without this, token 0 of sequence 2 would have a position embedding as if it were token N of the combined tensor, which is incorrect.

Effect on training loss

Because packing increases the number of tokens seen per training step (by eliminating padding waste), the per-step training lossThe loss is computed over all non-padding tokens in a batch. With packing, there are more real tokens per step, so the loss is computed over a larger, more varied sample — this makes it less noisy (less variable) step-to-step, but the absolute value is on a different scale than unpacked training. is computed over more tokens. The loss values are less variable than unpacked training (the gradient signal is based on more tokens per step), but the raw numbers differ. This is expected and correct — the model is simply learning more per step.

In past versions of Unsloth, increasing batch size beyond 32 made training slower due to padding overhead — each large batch needed to accommodate very long outlier sequences. With packing enabled, larger batch sizes are beneficial again: more sequences packed together = higher valid token density = faster training.

Chapter 7 — How to Enable Packing 8

7. How to Enable Packing

Step 1 — Update Unsloth

Padding-free batching is on by default in the latest version. Updating is sufficient to get the free 1.1–2× speedup and 30% VRAM reduction with zero code changes:

pip install --upgrade --force-reinstall --no-cache-dir --no-deps unsloth
pip install --upgrade --force-reinstall --no-cache-dir --no-deps unsloth_zoo

Step 2 (optional) — Enable explicit packing

For larger gains on short-sequence datasets, add packing = True to your SFTConfigThe configuration object for Unsloth's SFTTrainer (Supervised Fine-Tuning Trainer). It inherits from Hugging Face's TrainingArguments and adds Unsloth-specific settings like packing, max_length, and attention backend selection.:

from unsloth import FastLanguageModel
from trl import SFTTrainer, SFTConfig

model, tokenizer = FastLanguageModel.from_pretrained(
    "unsloth/Qwen3-14B",
)

trainer = SFTTrainer(
    model = model,
    processing_class = tokenizer,
    train_dataset = dataset,
    args = SFTConfig(
        per_device_train_batch_size = 1,
        max_length = 4096,
        packing = True,   # ← enables uncontaminated sample packing
    ),
)
trainer.train()
Note on loss numbers. With packing=True, the training loss values and the reported number of dataset rows will differ from an unpacked run — short sequences are merged, so the row count shrinks. This is expected. If you need loss numbers that are directly comparable to a non-packed baseline, use padding-free mode only (packing=False, default).
